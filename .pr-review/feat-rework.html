1. Problem

Three gaps existed in Carbon's production and planning pipeline:

  1. No rework path. When a job operation failed quality checks, there was no way to loop back to an earlier operation, clone the relevant steps, and re-execute them. Operators had to manually create new operations or work around the system.
  2. No demand-forecast lineage. MRP generated demand forecasts, but there was no way to trace why a forecast existed—which parent job, sales order, or demand projection caused it. Planning decisions lacked auditability.
  3. Purchasing planning friction. The purchasing planning flow needed improvements to handle supplier–period grouping, MOQ enforcement, existing PO reuse, and supply forecast deduplication.

3. Rework Flow (the headline feature)

3.1 User Journey

Operator on MES

  1. Working on operation "Inspect"
  2. Finds defect → opens Rework action
  3. ReworkModal loads upstream operations via BFS
  4. Selects target: "go back to Weld"
  5. Sets quantity: 2 units, enters reason
  6. Submits

Backend (single transaction)

  7. findReworkPath(): BFS finds path Weld → Paint → Inspect
  8. Creates rework record (reason, qty, traceability)
  9. Clones 3 operations: Weld(R), Paint(R), Inspect(R)
     • First op: status = "Ready", operationOrder = "With Previous"
     • Remaining: status = "Waiting"
  10. Clones steps, tools, parameters for each
  11. Wires DAG: Weld(R) → Paint(R) → Inspect(R)
  12. Convergence: downstream ops now also depend on Inspect(R)
  13. Records productionQuantity type = "Rework"

Post-transaction

  14. Triggers reschedule for date recalculation
  15. Operator sees new operations on their board

3.2 Before / After DAG

Before: linear

Weld → Paint → Inspect → Pack
                      ✘ defect found

After: branch & converge

Weld → Paint → Inspect →→→→ Pack
                  └ Weld(R) → Paint(R) → Inspect(R) ┘

3.4 Auto-Completion Trigger

The complete_rework_trigger in the migration fires when any jobOperation status changes to 'Done'. It checks if ALL operations sharing the same reworkId are Done, and if so, sets rework.completedAt.

migrations/20260527142837_rework.sql
+ CREATE OR REPLACE FUNCTION complete_rework()
+   RETURNS TRIGGER AS $$
+ BEGIN
+   IF NEW.status = 'Done' AND NEW."reworkId" IS NOT NULL THEN
+     IF NOT EXISTS (
+       SELECT 1 FROM "jobOperation"
+       WHERE "reworkId" = NEW."reworkId" AND status != 'Done'
+     ) THEN
+       UPDATE rework SET "completedAt" = now()
+       WHERE id = NEW."reworkId";
+     END IF;
+   END IF;
+   RETURN NEW;
+ END; $$ LANGUAGE plpgsql;

10. Reviewer Takeaway

Mental model: This PR adds a "branch and reconverge" primitive to job operations. When something goes wrong mid-production, the system clones the relevant operation path, runs it as a parallel branch in the DAG, and merges back downstream. Everything else—scheduling, MRP, quantity tracking—is made rework-aware without changing its core logic.
