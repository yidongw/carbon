## Findings considered and rejected

So nobody re-audits them:

- **Missing `useEffect` / `toast` imports** in `picking.$pickingListId.tsx`:
  FALSE — both are imported (`:15` `toast`, `:19` `useEffect`). Two subagents
  hallucinated this.
- **Redundant `isShort ? Pick : Pick` ternary** at `picking.$pickingListId.tsx:296`:
  FALSE — it is a single `<Trans>Pick</Trans>` button; no redundant ternary.
- **Empty `requirePermissions(request, {})` in MES routes/middleware**: BY
  DESIGN — the empty permission map is the established MES convention (the MES
  `userMiddleware` itself uses it). Tenant isolation is enforced via the RLS
  client / `companyId`, addressed by plan 001 for the service-role paths.
- **Single-column PK `("id")` instead of composite `("id","companyId")`**: this
  is the de-facto current convention (matches recent tables like `fixedAsset`);
  the convention doc is stale, not the migration.
- **NUMERIC(12,4) quantities instead of INTEGER**: fractional pick quantities are
  legitimate in manufacturing; the convention's "INTEGER for quantities" is a
  simplification. Not a defect.
- **`update_picking_list_status` trigger updates `pickingList` by id without a
  `companyId` filter**: `pickingList.id` is a unique PK, so no cross-tenant risk.
- **`getAssignedPickingLists` missing `companyId` filter**: runs only on the
  RLS-enforced client; defensible.
- **Missing composite `pickingListLine(companyId, status)` index** and
  **PickingKanban memoization**: speculative perf, no evidence of a real
  bottleneck at current scale. Not worth doing now.
